## Lesson package depth criteria (not field presence)

Each lesson must be a **teach → prove** package, not a flashcard. Automated gates (`validateLessonPackageDepth` / `validateCodingProgram`) fail CI when a lesson lacks real instructional depth.

| Block | Human review standard |
| --- | --- |
| `objective` / `whyItMatters` | One clear outcome + stake; not a slogan |
| `teach` | Multi-paragraph explanation with misconceptions called out |
| `workedExample` | Multi-step commands/code with expected shape—not a one-line motto |
| `tryThis` | Short practice intro plus numbered `tryThisSteps` checklist when present |
| `expectedOutput` / `hint` | Expected shape/output callout; optional revealable hint (not a credential grade). Required on early Day 1 Python packages (lessons 03–04) |
| `commonFailures` | ≥2 failure modes with recovery |
| `checkYourself` | ≥2 retrieval questions with revealable answers (not a credential grade) |
| `bridgeToLab` | Explicit Code Lab / Systems Lab link where a matching challenge exists (Day 1: Terminal escape + Equipment triage CLI) |
| `defensePrompt` | Oral-defense close that matches the lesson |

**Reviewer walkthrough:** complete Day 1 packages in the reader UI before treating Days 2–4 as approved tone/density. Labs remain the *prove*; lessons are the *teach*.
