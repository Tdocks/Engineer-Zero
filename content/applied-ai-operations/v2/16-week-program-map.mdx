# 18-Week Program Map

| Weeks | Focus | Required evidence |
| --- | --- | --- |
| 1–2 | System maps, Python functions, data, decisions, and errors | Explain a request path and independently write a small validated transformation |
| 3–4 | Files, tracebacks, JSON, logs, HTTP, FastAPI, and validation | Diagnose a fictional failure and build a read-only validated endpoint |
| 5–6 | SQL, Git, pytest, identity, authorization, and AI-assisted review | Tested request path, access-control evidence, and reviewed generated diff |
| 7–8 | Discovery, suitability triage, prompts, structured outputs | Workflow map, no-AI comparison, schema boundary |
| 9–10 | Retrieval, metadata, citations, tools/workflows | Permission-aware retrieval design and action-permission matrix |
| 11–12 | Evaluation, failure taxonomy, model selection, security | Golden set, release gate, threat model, and regression decision |
| 13–14 | Observability, deployment, incident recovery, and technical leadership | Audit design, degraded-service runbook, and stakeholder decision record |
| 15–16 | Reliability, red-team review, multi-team delivery | Recovery exercise, remediation record, and decision record |
| 17–18 | Capstone, executive recommendation, project defense | Two prototypes, capstone package, and technical defense |

## Prerequisite rules

- A learner must complete all six Foundation Bridge **Prove** activities before the production prototype exercises.
- Retrieval work requires successful identity/authorization and structured-output activities.
- Tool action design requires evaluation, policy, approval, and idempotency evidence.
- Rust is optional and only unlocks after Python service and evaluation evidence are complete.

## Capstone definition

The capstone is a decision package, not a demo: workflow map; AI-suitability decision; data/authorization boundary; architecture; prototype scope; evaluation plan; risk register; audit/approval design; rollout/rollback plan; executive recommendation; and recorded technical defense.
