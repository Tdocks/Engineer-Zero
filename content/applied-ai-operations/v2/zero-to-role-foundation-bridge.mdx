# Zero-to-Role Foundation Bridge

This six-week bridge is for learners who do not yet have dependable programming or application-system fluency. It does not rush a learner into AI architecture vocabulary before they can reason about an ordinary request, data boundary, test, error, and reviewable code change.

## Evidence standard

| Level | Learner can do | This does not claim |
| --- | --- | --- |
| Know | Explain a concept, identify its operational impact, and name the accountable specialist. | Configuration, administration, or independent production operation. |
| Practice | Complete a bounded guided task with help, then explain the decision and limitation. | Independent production implementation. |
| Prove | Independently build, troubleshoot, or defend saved evidence against a deterministic rubric. | Organizational authorization to deploy, access real data, or bypass review. |

## Six-week sequence

| Week | Focus | Prove evidence |
| --- | --- | --- |
| 1 | How software systems move work | A clear client → service → data → identity → logs system map. |
| 2 | Python reasoning | A small data transformation with explicit inputs, decisions, and error handling. |
| 3 | Developer workflow | A traceback-based diagnosis with a bounded repair hypothesis and evidence. |
| 4 | APIs and validation | A read-only FastAPI endpoint with validation, error behavior, and tests. |
| 5 | Data, tests, identity, and change | Relational evidence model, access-control tests, and reviewable Git explanation. |
| 6 | AI-native engineering | A reviewed and repaired AI-assisted Python change with honest ownership disclosure. |

## Gate to local prototypes

The two fictional local prototype repositories require all six **Prove** activities. The goal is not to create artificial friction; it ensures a learner can read a traceback, test an API boundary, explain authorization, and review AI-generated changes before they attempt a larger service.

## Sources and instructional use

Every activity in the typed catalog names an official source, the exact claim it supports, a locator, source version, accessed date, and a revalidation date. Sources explain technical facts; the course’s original scenarios, exercises, and rubrics teach how to apply those facts in fictional enterprise conditions.

The instructional pattern—worked examples, retrieval checks, active practice, feedback, and revision—is separately mapped to [Dunlosky et al. (2013)](https://doi.org/10.1177/1529100612453266) and [Freeman et al. (2014)](https://doi.org/10.1073/pnas.1319030111). Those studies support the course format; they do not certify that completing an activity proves job competence.
